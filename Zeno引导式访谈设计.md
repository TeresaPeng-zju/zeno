# Zeno 引导式访谈设计

> 首批用户：开发者 / 产品经理。目标：用"像聊天一样"的低摩擦体验，采集出"喂得动迁移引擎"的高密度结构化数据。

---

## 一、三条设计原则

1. **对话是采集器，不是产品。** 每一轮 AI 都在做两件事：往槽位里填值、对模糊回答追问证据。聊天体验只是外壳，目的永远是把 Skill Graph 填满。
2. **要证据，不要标签。** 不收集"会 Python"，收集"用 Python 做了什么、什么决策、什么结果、什么指标"。标签产出通用建议，证据产出迁移洞察。
3. **先锁目标，再挖来源，最后按差距深挖。** 没有目标岗位，就无法判断哪些经验值得深挖。访谈顺序本身就是一种筛选。

---

## 二、槽位结构（Slot Schema）

迁移引擎需要的数据模型。每个槽位标注：是否必填、"填满"的判定标准。

### A. 目标层（去哪里）—— 必须最先锁定
| 槽位 | 说明 | 填满标准 |
|---|---|---|
| `target_role` | 具体岗位，非泛指 | "AI 产品经理"而非"AI 方向"；能对应一个真实 JD |
| `target_seniority` | 目标级别 | 初级/资深/带团队 |
| `target_context` | 行业/公司类型 | 大厂 vs 创业、To B vs To C 等 |
| `motivation` | 为什么转、为什么现在 | 有具体触发事件，而非"想搞 AI" |
| `constraints` | 每周可投入时长、deadline、经济缓冲 | 量化：如"每周 10h，6 个月内，有 1 年积蓄" |

### B. 来源层（从哪来）—— 重纹理
| 槽位 | 说明 | 填满标准 |
|---|---|---|
| `current_role` | 现岗位 + 年限 | — |
| `domain_expertise` | 可携带的行业认知 | 具体领域，如"供应链 SaaS 三年" |
| `spike_achievements` | 2–3 个最自豪的项目 | 每个含：你的角色、关键决策、结果、可量化指标 |
| `hard_skills[]` | 硬技能 + 证据 + 自评水位 | 每项有一句证据，水位待校准 |
| `meta_skills[]` | 软/元技能（沟通、应对模糊、利益相关方、领导力） | 用故事佐证，非自我标榜 |
| `portable_knowledge` | 可迁移的领域知识/方法论 | — |
| `artifacts` | 可展示的作品/作品集 | 有则记链接，无则标记为缺口 |

### C. 迁移探针层（按目标自适应生成）—— 差异化核心
> 由"来源 → 目标"之间**最不确定的迁移边**动态生成。下表为 `开发者 → AI PM` 的示例探针。

| 探针 | 为什么问 | 填满标准 |
|---|---|---|
| `product_sense` | 工程师转 PM 最大不确定边 | 有"为何做/不做某功能"的判断案例 |
| `user_empathy` | PM 核心，工程背景常缺 | 有直接接触用户/改需求的实例 |
| `prioritization` | 资源取舍能力 | 有一次真实砍需求/排序的决策 |
| `written_comms` | PM 高频且可证伪 | 写过 PRD/提案/复盘，可举证 |
| `data_literacy` | AI PM 必备，工程背景是优势 | 能说出用过的指标和据此做的决策 |

### D. 计划约束层（怎么定方案）
| 槽位 | 说明 |
|---|---|
| `learning_style` | 偏看/偏做/偏带教 |
| `risk_tolerance` | 激进换岗 vs 稳健补能力 |
| `goal_type` | 要一份工作 vs 要一套能力 |
| `market_context` | 地域/目标市场 |

### E. 校准层（客观评估）
对每个 `hard_skills` / 探针：先让用户自评，再用追问逼出证据，最后由引擎给出**校准后水位**（可能高于或低于自评）。这是 Zeno 对抗"大模型一味奉承"的关键。

## 三、对话脚本（分阶段）

每阶段标注：AI 意图 · 开场话术 · 追问规则 · 填充槽位。

### Phase 0 · 锁定目标（2–3 轮）
- **意图**：拿到具体目标岗位、动机、硬约束。
- **开场**："在我帮你分析之前，先确认方向——你想转的具体是什么岗位？如果已经看上了某个 JD，直接发我最好。"
- **追问规则**：若目标泛化（"AI 方向"）→ 逼具体（"是偏产品、偏工程还是偏运营？面向 To B 还是 To C？"）。
- **填充**：`target_*`, `motivation`, `constraints`

### Phase 1 · 挖出能力高峰（开放、讲故事）
- **意图**：用故事捕捉证据，发现可迁移强项。这是"惊艳时刻"的素材来源。
- **开场**："抛开岗位，先说说你职业生涯里最自豪的一两件事——做了什么、你具体负责哪块、最后结果怎样？"
- **追问规则**：故事缺少【决策】或【指标】→ 追问一次（"那个节点上是你拍的板吗？凭什么这么判断？""有数字能体现效果吗？"）。
- **填充**：`spike_achievements`, `meta_skills`, `hard_skills`(部分)

### Phase 2 · 定向迁移探针（自适应深挖）
- **意图**：只深挖这次转型最不确定的几条迁移边。
- **开场**（动态）："你要转 AI PM，工程底子是加分项。但有几块我得确认一下——比如，你有没有过为了某个目标，主动砍掉一个技术上很想做的功能的经历？"
- **追问规则**：每条探针最多追问 2 次；拿到一个具体实例即停，避免疲劳。
- **填充**：Phase C 全部探针

### Phase 3 · 校准与客观评估
- **意图**：对自评做证据校准，敢于指出虚高。
- **开场**："你把 SQL 自评成'熟练'，能说一个你靠它自己做出某个判断的场景吗？"
- **追问规则**：证据不足 → 引擎下调水位并标注"待验证"，而非顺着用户。
- **填充**：`hard_skills` 校准值、`E` 层

### Phase 4 · 计划约束
- **意图**：补齐生成方案所需的偏好与边界。
- **开场**："最后几个快问快答：每周大概能挤出多少小时？你更想先拿到一份 offer，还是先把能力补扎实？"
- **填充**：`D` 层

### Phase 5 · 回显确认（建立信任 + 补漏）
- **意图**：把抽取出的画像回显给用户，让其纠正。纠正动作本身又能补全槽位。
- **话术**："我先把听到的整理给你看——【可迁移强项 3 条 / 待补缺口 3 条 / 校准后的真实水位】。哪里不对，直接改。"

## 四、追问规则引擎（让对话显得"懂行"的逻辑）

```
对每条用户回答：
  IF 槽位所需的【决策 / 指标 / 具体实例】缺失:
     → 发起 1 次定向追问（针对缺失的那一项，不泛问）
  IF 已追问次数 >= 该话题上限(默认2):
     → 标记槽位为"证据不足"，停止追问，进入下一槽位
  IF 槽位已有足够证据:
     → 不追问，前进
全局:
  - 总轮次软上限（如 25 轮），接近时优先填必填槽位
  - 任意时刻用户可说"跳过"，该槽位标记为空并在回显阶段提示
```

要点：追问要**针对缺失的那一项**（缺指标就只问指标），这种精准追问正是和"通用大模型泛泛而谈"拉开差距的体感来源。

## 五、样例片段（开发者 → AI PM）

> **Zeno**：在分析前先锁方向——你想转的具体是哪种 AI 岗位？有看中的 JD 直接发我。
> **用户**：AI 产品经理，偏 To B。
> **Zeno**：好。先抛开岗位——你职业里最自豪的一件事是什么？
> **用户**：主导重构了订单系统。
> **Zeno**：这个"主导"具体到什么程度——架构是你定的吗？过程中最关键的一个取舍是什么？*(追问：缺决策)*
> **用户**：是我定的，当时砍掉了一个实时同步方案，因为成本扛不住，改成准实时。
> **Zeno**：这其实就是产品判断——在成本和体验间做取舍，并为业务结果负责。这块经验能直接迁到 AI PM。有数字能体现重构的效果吗？*(回扣迁移 + 追问指标)*

—— 这一段里，"砍方案"的故事同时填充了 `spike_achievements`、`prioritization` 探针、`product_sense`，并完成了一次"你比想象中更接近"的迁移回扣。
